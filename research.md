# Contact Enrichment — Research

## Current Contact System

### Data Model

A contact is stored in two places:

1. **`.corky.toml`** — config entry under `[contacts.{name}]`:
   ```toml
   [contacts.alice]
   emails = ["alice@example.com", "alice@work.com"]
   labels = ["correspondence"]
   account = "personal"
   ```
   All three fields are optional and default to empty.

2. **`contacts/{name}/`** — directory with context files:
   ```
   contacts/{name}/
     AGENTS.md          # Scaffolded template
     CLAUDE.md -> AGENTS.md
   ```

### Contact struct (`src/config/contact.rs`)

```rust
pub struct Contact {
    pub emails: Vec<String>,   // #[serde(default)]
    pub labels: Vec<String>,   // #[serde(default)]
    pub account: String,       // #[serde(default)]
}
```

Key functions:
- `load_contacts(path: Option<&Path>) -> Result<BTreeMap<String, Contact>>`
- `save_contact(name: &str, contact: &Contact, path: Option<&Path>) -> Result<()>`

### Creation Flow (`src/contact/add.rs`)

`run(name, emails, labels, account)`:
1. Bail if contact exists in `.corky.toml` or directory exists
2. Create `contacts/{name}/` with AGENTS.md template + CLAUDE.md symlink
3. `save_contact()` via `toml_edit` (format-preserving)
4. Optionally add labels to account sync config

The AGENTS.md template has placeholder sections: Relationship, Tone, Topics, Notes.
Generated by `generate_agents_md(name)` (private function).

### CLI Structure

Currently a flat command: `corky contact-add NAME --email EMAIL`.
Defined as `Commands::ContactAdd` in `src/cli.rs:97-112`.
`--email` is required. No way to create a contact without knowing the email upfront.

Other subcommand groups for reference: `Draft(DraftCommands)`,
`Mailbox(MailboxCommands)`, `Sync { command: Option<SyncCommands> }`.

### Manifest Integration

`src/sync/manifest.rs:generate_manifest()` runs after sync:
1. Load all `[contacts]` from `.corky.toml`
2. Build `email->contact-name` lookup (case-insensitive)
3. Scan `conversations/*.md`, parse each thread
4. For each message, extract email via `<([^>]+)>` regex on `from` field
5. Match against lookup, write `contacts = ["name"]` per thread entry

Contacts only appear in manifest.toml if they have email addresses configured.

## Conversation Format

Message headers: `## {Sender Name} <{email}> \u{2014} {date}`

Parsed by `MSG_HEADER_RE`: `^## (.+?) \u{2014} (.+)$`

The `from` field includes display name and email: `Alice Smith <alice@example.com>`.
Email extracted by `EMAIL_RE`: `<([^>]+)>`.
Display name is everything before ` <email>`.

Metadata lines: `**Labels**: label1, label2` / `**Accounts**: acct1`

## Owner Identification

Owner accounts live in `[accounts.*.user]` in `.corky.toml`:
```toml
[accounts.personal]
user = "brian@example.com"
```

To filter owner senders from a conversation, collect all `accounts.*.user`
values and compare against sender emails. The `CorkyConfig` struct
(`src/config/corky_config.rs`) has `accounts: HashMap<String, Account>`,
where `Account.user` is the email address.

## Conversation Search Paths

Conversations can live in:
- `{data_dir}/conversations/{slug}.md` — root conversations
- `{data_dir}/mailboxes/{name}/conversations/{slug}.md` — mailbox conversations

Finding by slug requires searching both locations.

## Data Extraction Opportunities

From a single conversation file we can extract:

| Data | Source | Use |
|------|--------|-----|
| Email addresses | `<email>` in `## From` headers | Contact emails field |
| Display name | Text before `<email>` | Contact name (slugified) |
| Labels | `**Labels**:` metadata line | Contact labels field |
| Accounts | `**Accounts**:` metadata line | Contact account field |
| Subject | `# Subject` heading | Pre-fill Topics section |
| Thread activity | Last message date, message count | Info display |

## Manifest as Index for `contact info`

`manifest.toml` already tracks `contacts = ["name"]` per thread:
```toml
[threads.{slug}]
subject = "Thread Subject"
contacts = ["alice"]
labels = ["label1"]
last_updated = "RFC 2822 date"
```

For `contact info`, we can scan manifest entries where `contacts` includes
the given name — no need to re-parse every conversation file.

Manifest lives at `{data_dir}/manifest.toml`. For mailboxes, each has its own
manifest at `mailboxes/{name}/manifest.toml`.
